# 顺便一提乐观锁与悲观锁

这是乐观锁的一种实现方式，通过在 update 时校验旧的 balance 值来保证并发安全，属于基于 CAS 思想的条件更新，但不是 MyBatisPlus 内置的 @Version 标准乐观锁。

乐观锁和悲观锁都是并发控制手段。乐观锁与悲观锁都是基于一种假设，乐观锁假设冲突很少，更新时才检查数据有没有被别人改过，悲观锁假设冲突一定会发生，操作数据前先把它锁住。

悲观锁假设并发冲突一定发生，在操作数据前先加锁，其他线程必须等待。在高并发、读多写少的场景下一般使用乐观锁，在强一致、冲突频繁的场景才使用悲观锁。

# 服务拆分原则

什么时候需要把单体架构拆分为微服务？创业型项目建议先用单体架构，快速开发快速试错，随着规模扩大逐步拆分。已确定的大型项目推荐直接使用微服务架构。

从拆分角度来讲尽可能的做到单一职责，高内聚，低耦合。横向拆分：抽取公共类，提高复用性。纵向拆分：按照业务模块拆分。

服务被拆分后，每一个都是一个项目，或者说是模块。

# Gateway 网关

网关主要解决的是，当微服务向外暴露不同的端口时，前端该如何请求？登录后，用户信息与校验如何共享？能否保证安全？

当我们使用网关时，前端无需做出其他修改，直接请求 8080 即可，网关会为你自动路由，并且网关也会为你进行负载均衡。当然我们的网关也是会上注册中心的，当微服务出现地址变更的时候，注册中心会为网关推送最新的地址。网关也会为我们进行身份校验，校验通过后，当我们要请求深层的微服务时，网关还会带着你的身份信息进行向下传递。网关的存在，方便的前端的请求，也保护了后端的微服务。

既然网关会上注册中心，因此网关本身也是一个独立的微服务。

  1. 根据我们之前所说的，网关首先需要一次全局过滤，因为所有的请求先打到网关，那么其实一些不需要权限验证的路径我们完全可以选择跳过，比如登录接口如果不是那种接口的话，我们就进行下一步操作。

  2. 前端往往会给我们在请求头的 Authorization 中加入后端返回的 Token 信息，我们需要对他进行一次截获，然后解析，如果通过 JwtTool 成功解析的话那就说明 Token 有效，可以继续进行下一步操作，否则就需要标记为未授权，并直接终止其请求。

  3. 那么到了这里，说明网关已经成功将这次请求视为已授权请求，他需要向下告知其他微服务，在这里就是通过 exchange.mutate 在请求头中塞入 user-info，当然可以是结构性数据，也可以是纯粹的用户唯一标识。exchange.mutate 是一次链式编程，他会返回一个全新的 ServerWebExchange，我们需要将这个 swe 通过 chain.filter (swe) 返回。

  4. 好那么下一步就是，其他的微服务怎么获取到这个 userInfo 信息？首先他在请求头中，我们可以设置一个统一的拦截器来获取。这个 preHandle 只做一件事，那就是检查能否从请求头中获取到 user-info，如果有就使用 UserContext 将其存到 ThreadLocal 之中。

  5. 那么在请求完成之后要记得清理 ThreadLocal。

  6. 拦截器设置完成之后，我们需要把这个拦截器注册到容器里，我们使用 MvcConfig 继承 WebMvcConfigurer 通过 registry.addInterceptor 注册即可。

  7. 这里我们还设置了：@ConditionalOnClass (DispatcherServlet.class)，这是因为在微服务的网关中，没有 Mvc 的依赖，所以你设置 Mvc 有关的配置，网关是会报错的，因此我们通过 DispatcherServlet.class 来使得其不在网关中生效。

  8. 然而这个时候，MvcConfig 并没有生效，因为 MvcConfig 在 common 模块中，而 Spring Boot 默认不会扫描依赖 Jar 里的 @Configuration，所以需要通过 EnableAutoConfiguration 显式让 Spring Boot 加载它。

为什么 Spring Cloud 网关要用 GlobalFilter，而微服务要用 HandlerInterceptor？

Spring Cloud Gateway 使用 GlobalFilter 是因为它基于 WebFlux 的响应式编程模型，而微服务通常基于 Spring MVC 的 Servlet 模型。WebFlux 没有 DispatcherServlet 和 HandlerInterceptor 机制，因此需要使用 GlobalFilter 进行全局拦截；而 Spring MVC 依赖 DispatcherServlet 调度请求，所以通过 HandlerInterceptor 实现业务层拦截。

# 用户信息如何在微服务间进行传递（OpenFeign）

OpenFeign 的调用，不同于 Gateway，Gateway 在每次独立的请求中通过拦截器向下传递用户信息，但这种传递是单次的，因为如果我们的业务复杂度高，势必会出现微服务于微服务之间的调用，此时这种之间的调用那一定不是由 Gateway 网关完成的，每次微服务间的调用是由 OpenFeign 实现的，然而我们并没有为 OpenFeign 设置有关拦截处理，因此无法从 UserContext 中获取到相应的 UID。

ThreadLocal 是线程级别的数据存储，只能在同一个线程中共享。在 trade-service 中调用 OpenFeign 时，Feign 拦截器可以在当前线程中，从 ThreadLocal（UserContext）获取用户信息，并将其写入 HTTP Header。当请求到达 cart-service 时，cart-service 再通过拦截器从 Header 中读取，用户信息并重新放入 ThreadLocal，从而在当前服务内部实现数据共享。

同一个请求共享 ThreadLocal ，当微服务发起一个新请求时，那里是一个全新的 ThreadLocal ，两个 ThreadLocal 相互独立。

ThreadLocal 只能在同一线程中共享数据。微服务之间调用（例如 trade 调 cart）实际上是新的 HTTP 请求，ThreadLocal 不会自动传递。因此需要通过 Feign 拦截器把 ThreadLocal 中的数据写入请求 Header，再在下游服务中，通过拦截器读取 Header 并重新放入 ThreadLocal。

# 配置管理 —— 微服务配置的 “预制菜”

  • 每个服务都有自己的配置（数据库连接、消息队列、API 地址、限流策略等）
  • 服务数量可能几十、几百
  • 配置可能需要动态调整（灰度发布、限流、开关功能）

如果每个服务都单独管理配置：

  1. 运维压力大：配置分散在各个服务，改一个参数要登几十台服务器。
  2. 不一致风险：可能不同环境、不同服务配置不一致。
  3. 不支持动态变更：服务已经启动了，改配置还需要重启服务。

配置管理就是把原本分散、静态、易出错的配置集中、动态、可管理化，解决微服务环境下的维护和运维痛点。

# 雪崩问题

在微服务架构中，“雪崩效应” 是一种因级联故障导致整个系统瘫痪的现象。简单来说，就是底层服务的微小故障，经过层层传递和放大，最终让原本健康的高层服务也随之崩溃。

# 如何解决

微服务保护的方案有很多，比如：

  • 请求限流：流量整形，保护服务的提供者。（提供者）
  • 线程隔离：限定每一个业务使用的服务数量，服务的请求者有有限的线程池。类似于船舱模型。（调用者）
  • 服务熔断：断路器统计请求异常比例，超出异常阈值，就会熔断这个业务。失败时执行 fallback 兜底逻辑。（调用者）

# Seata——XA 模式

基于数据库层面的 XA 协议支持，主要强一致性保证，并对业务代码无侵入，但是它的性能相对较低。

# Seata——AT 模式

AT 模式是 Seata 主推的模式，避免了等待时间过长的问题，那么我们来看看他是怎么个结构。

收到请求，更新之后立即提交，并保存更新前的快照，如果大家都没事就删除快照，出问题那就回恢复快照。

# XA 与 AT 模式最大区别是什么？

  • XA 模式一阶段不提交事务，锁定资源； AT 模式一阶段直接提交，不锁定资源。
  • XA 模式依赖数据库机制实现回滚； AT 模式利用数据快照实现数据回滚。
  • XA 模式强一致； AT 模式最终一致

可见，AT 模式使用起来更加简单，无业务侵入，性能更好。因此企业 90% 的分布式事务都可以用 AT 模式来解决。

Work queues，任务模型。简单来说就是让多个消费者绑定到一个队列，共同消费队列中的消息。

Work 模型的使用：

  • 多个消费者绑定到一个队列，同一条消息只会被一个消费者处理
  • 通过设置 prefetch 来控制消费者预取的消息数量

# 交换机

交换机的类型有四种：

  • Fanout：广播，将消息交给所有绑定到交换机的队列。我们最早在控制台使用的正是 Fanout 交换机
  • Direct：订阅，基于 RoutingKey（路由 key）发送给订阅了消息的队列
  • Topic：通配符订阅，与 Direct 类似，只不过 RoutingKey 可以使用通配符
  • Headers：头匹配，基于 MQ 的消息头匹配，用的

# Fanout 广播交换机

Exchange 只负责转发消息，不具备存储消息的能力，因此如果没有任何队列与 Exchange 绑定，或者没有符合路由规则的队列，那么消息会丢失！
